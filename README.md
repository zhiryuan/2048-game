# 2048-game
### 使用环境
```
python 3.9<=ver<3.13
pygame 2.6.1
```
注意pygame目前尚不支持python>=3.13的较新版本
### 简介

可能是作为数算课的大作业？

本项目通过手工编程，用python以及pygame图形库实现2048，通过OOP描述界面上每个矩形对象的信息，实现动画效果。

由于用OOP的方式，可以比较灵活地在不做大量修改的情况下新加入一些逻辑，有比较好的可扩展性。
手工编程让我能对如何写一个比较复杂的程序有更好的理解和掌握，虽然代码写来写去，可能看起来很乱，也有不少冗余。可能之后向AI学习一下如何更好地实现这些逻辑。

除了`2`, `4`还有一定概率生成`×2`，`×4`，`×0`以及`#`

两个`#`可以合成`×2`，两个`×2`可以合成`×4`，同时获得大量分数

实现的动画效果有移动时的平移、生成新方块的放大、以及加分的平移+淡出

### 一些记录
作为古法编程enthusiast，在数算B课上看到一道模拟2048的题目，觉得很有意思，希望能不借助AI Coding自己写一写。

其实我小学时用c++写过几次命令行界面的2048，还记得那时只会用Dev-c++，有一次在学校微机课上划水，用70行就写好过一个。

但是之前一直只会控制台输入输出，c++借助windows.h的UI写法复杂且无法跨平台。
现在既然pygame已经是一个比较易用的图形化界面库，那就可以用它还原一下网页上html写的制作精美的2048，还可以往里面加各种~~奇怪的东西~~

倘若单纯地将输入改为键盘输入，将输出改为窗口绘制图形，未免太容易了些，仅需写好控制台式2048后自己写一个输入、输出函数即可。

然而，网页上2048每次移动时都有很丝滑的动画效果，仅凭在循环中监测键盘输入→修改二维数组→图形化显示的逻辑，是做不到这样的。

这需要自己处理一个时序逻辑，在循环中每次动画播放一帧，同时读取输入。
这时候，在模拟移动过程时，既要考虑数组怎么变化，又要考虑在当前状态下调整将要播放的动画。

每个数是存储在数组中还是直接存储在动画播放时每个“块”中？要是都有的话不是会有冗余吗？
要是读入过快、快于动画播放速度应该怎么处理？应该直接跳过吗？

思索良久，我决定做一个扩展性比较强的东西，这样方便我往里面改来改去，并且自己加一些东西。
询问AI得知处理数据逻辑时向显示的部分传出一个队列，队列表示要显示的动画列表会好一些。
于是我希望将预定的动画用类似`('mov', (a, b), (c, d))`的方式存储下来：存储预定动画并传出，比直接在移动的两层循环里设置动画更好，
是因为这样我只需要在处理数据的时候只处理数据，在处理动画时只处理动画，每层只用考虑该层的事情，可以减少很多思维负担；
同时我可以根据当前队列长度来调整播放速度，而且如果想修改逻辑，甚至说复用2048的UI实现“数字华容道”，我可以只修改一小部分核心代码。

（但是后来写的时候发现动画是同时显示的，并且进行一次操作后希望要加入先后两个时间的动画：①移动；②数值翻倍或加入新block。
我最后是通过队列套队列，内部队列存储当前时刻的所有动画，外部按更新时间的顺序，来存储预定动画。
不过由于动画和数据不同步，屏幕上的块数量会多于当前非零的格子数量，所以用`('mov', start, end)`表示时，每个块对应一个位置，就要考虑他们何时对应什么位置）

第一次尝试时，我先从数据处理层面写核心逻辑代码，复用之前的`T20052.py`的代码，在上面加上表示动画的队列；
但是由于动画表示要用到原始的坐标，同时我希望2048网格的大小能设置成m×n，转来转去很麻烦。
经过一些尝试后，把写的`2048game.py`弃坑了。

于是，我打算从我不太熟悉的UI部分写起，先搞清楚UI是怎么一回事，再去写比较适配的核心逻辑。所以叫做`2048game_2.py`。
这次进展比较顺利，经过一些时间的搜索了解到pygame如何用`Surface`记录图形、如何处理字体，然后就开始写，
先想好了一个大致的框架：我得先实现一个block类，存储每个方块的坐标、大小、颜色；
同时方块里还有字，还要记录字体、字体大小、字体颜色，这样便于让这些block动来动去，并且与网格的不同位置绑定与解绑；
然后要记录他们当前的动画和状态；最后为了美化外观要记录矩形圆角半径；然后每个都有自己的`Surface`。
我既可以用这个类表示格子，也可以用它排版，绘制背景。后来我觉得block名字不好听，太泛用了，改成叫itembox。
不久我搜集了配色，写好了画背景的代码。

后来，又经过几天写来写去，最后竟然能跑了（调试时主要是方块绑定在哪个地方出问题）。
一开始运行很卡，以为是python很慢，后来发现是误将`clock.tick(x)`里的参数理解成延迟x毫秒（实际是类似于1000/x毫秒）
改完后就流畅了很多。
